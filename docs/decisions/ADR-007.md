# Decision: Knowledge Graph Data Model
## ADR-007

We decided to model repository memory as a typed Knowledge Graph.

Context:
To answer questions like "Why does this exist?" or "What breaks if I change this?", we need a data structure that explicitly captures relationships between different types of artifacts. Relational models are too rigid for arbitrary dependency chains.

Decision:
The model consists of `KnowledgeNode` (id, node_type, name, properties, created_at) and `KnowledgeEdge` (id, source_id, target_id, edge_type, confidence, created_at, properties). Node types include Requirement, Decision, Architecture, Evidence, Owner, CodeArtifact. Edge types include ImplementedBy, Drives, ValidatedBy, SupportedBy, ResultsIn, OwnedBy, DependsOn.

Consequences:
- Highly flexible schema that can evolve without database migrations.
- Uniform querying logic for all types of relationships.
- Easy to serialize to JSON for MCP tools.

This decision satisfies REQ-MEMORY-001.
This decision drives crates/ares-knowledge-graph/src/models.rs.

Status: Accepted
