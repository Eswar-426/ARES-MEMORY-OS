# Decision: Streaming Batch Ingestion
## ADR-003

We decided to use a streaming batch architecture for repository ingestion instead of loading the full graph into memory.

Context:
Initial ingestion loaded all nodes and edges into a Vec<GraphEvent> before persisting. For large repositories (Next.js: 106k files), this caused peak RSS to exceed 173 MB, violating the < 100 MB target.

Decision:
Replace Vec accumulation with a FnMut(GraphEvent) sink closure. Buffer events in batches of 2500 and commit each batch as a single SQLite transaction. This bounds memory to O(batch_size) instead of O(repository_size).

Consequences:
- Next.js RSS reduced from 173 MB to 71.2 MB.
- Memory usage is bounded by batch size, not repository size.
- Rollback safety: failed batches do not corrupt previously committed data.
- Ingestion time reduced from 106s to 21.9s due to fewer allocations.

This decision satisfies REQ-MEMORY-004 and REQ-INGEST-005.
This decision drives crates/ares-ingestion/src/graph.rs and crates/ares-knowledge-graph/src/store.rs.

Status: Accepted
