# Decision: SQLite as Knowledge Graph Store
## ADR-002

We decided to use SQLite (via rusqlite) as the persistent store for the ARES Knowledge Graph.

Context:
ARES requires a local-first, zero-configuration data store that supports ACID transactions, foreign key constraints, and WAL journal mode for concurrent read access. Graph databases (Neo4j, DGraph) were rejected due to deployment complexity. PostgreSQL was rejected due to external service dependency.

Decision:
Use SQLite with PRAGMA foreign_keys = ON and journal_mode = WAL. Store nodes in graph_entities and edges in graph_relationships tables.

Consequences:
- Zero external dependencies for the data layer.
- Single-file database (.ares/ares.db) simplifies backup and portability.
- WAL mode enables concurrent reads during ingestion.
- Referential integrity enforced at the storage layer, not the application layer.

This decision satisfies REQ-MEMORY-001.
This decision drives crates/ares-knowledge-graph/src/store.rs and crates/ares-store/src/lib.rs.

Status: Accepted
